My Journey

This project began on a road trip, where I discovered the joy of generating SQL queries. It was fun, but after a few months, I set it aside. Recently, a colleague introduced me to the Phoenix Framework, and I was instantly intrigued. Its extensive built-in features felt like a breath of fresh air, crafted by developers tired of repetitive web development challenges. I dove into the startup documentation and gained a solid foundational understandingâ€”not just of Phoenix, but of web design as a whole and its best practices. Much of this content was new to me, and it sparked a desire to create something of my own using these insights.

However, I faced a few obstacles:

  • I appreciate what I could learn from Elixir and the new problem-solving approaches it offers (i.e. functional paradigm), but it's not a skill that directly translates to my current work environment.
  • While I find Elixir's atoms and macros fascinating, I struggle to "just trust" them, feeling that it was too arbitrary.
  • I wanted to shift away from JavaScript in my free time, and the lack of a strict type system in Elixir is similar to JavaScript. For example, I spent a frustrating 30 minutes debugging xyzHtml versus xyzHTML.

Despite these challenges, my interest in the Phoenix Framework remains, particularly in its approach to web design and problem-solving. Ultimately, I aim to create something unique that reflects the knowledge I've gained.

Development Stage

Please note that this application is currently in early development. Many outputs are untested and primarily serve as proof of concept. The current functionality is based on my limited syntax knowledge and quick ideas, which means it's prone to errors and lacks detail.

This project is continuously evolving, with updates approximately every week.

Much of the current work is exploratory, focused on getting the application up and running. I plan to perform a cleanup pass soon.

Restrictions & Removals

  • Endpoint generation is currently limited to single primary keys. For optimal results, please use the + option with the table, as it helps in creating a RESTful API.
  • I have paused attempts at join logic while I focus on more straightforward implementations.
  • NodeJS Express endpoints have been temporarily removed as I prioritize development on the Go API.
  • The "read options" feature has been set aside in favor of a full read functionality for now.

Go API

This project aims to develop a robust application boilerplate to facilitate larger projects. Initial manual testing indicates that it works well with simple tables, though I still need to address some missing imports. 

    cd cmd/mysite/
    go get
    go vet
    # fix issues
    go run .

Pros

The clear organization of the Go app structure facilitates easy navigation and efficient code maintenance. Code reuse across different components enhances productivity by saving time during feature development. This setup supports scalability, allowing for the addition of new functionalities without disrupting existing code.

Cons

Despite its advantages, the structure can sometimes feel excessive for smaller projects, leading to unnecessary complexity. There is also a learning curve associated with fully grasping the nuances of this setup. Furthermore, the potential for overengineering exists, as simpler solutions may be overlooked in favor of more complicated ones, which can hinder efficiency.

Example Breakdown

Here's a quick overview:

  • /cmd/mysite is the entry point, specifically main.go.
  • /pkg contains reusable libraries:
    • /models/user.go defines the user model.
    • /services/user.go contains business logic related to users.
    • /repositories/user.go manages database interactions for user data.
  • /internal houses internal application logic:
    • /config for application configuration:
      • app.go for application-level settings.
      • db.go for database configurations.
    • /handlers for HTTP endpoints:
      • /user user related handlers
        • form.go handles form submissions for user data.
        • html.go handles the server-side rendering of user pages.
        • json.go handles JSON endpoints for user data.
    • /middleware is not yet in use but contains example.go as a placeholder for future middleware (e.g., authentication).
    • /routes defines application routes in routes.go.
  • /web contains frontend resources:
    • /templates for HTML templates:
      • /user user-related templates:
        • show displays single user details.
        • new includes a form for creating a new user.
        • index displays all users in a table.
        • edit features a form for editing user details.
    • /static contains static files (CSS, JS).
