My Journey

This project kicked off during a road trip when I discovered how enjoyable it is to generate SQL queries. I had a great time with it, but eventually set it aside for a few months. Then, a colleague introduced me to the Phoenix Framework, and I was immediately intrigued. Its built-in features felt like a refreshing change, designed by developers who were tired of the same old web development hassles. I dove into the documentation and gained a solid grasp of both Phoenix and general web design best practices. A lot of this was new to me, and it sparked my desire to create something unique using what I'd learned.

Why not just use Phoenix?

  • While I'm excited about what I can learn from Elixir and its new problem-solving methods (like the functional paradigm), the language is not used where I work.
  • Elixir's atoms and macros are fascinating, but I struggle to fully trust them; they can feel a bit arbitrary.
  • I wanted to step away from JavaScript in my free time, but the lack of a strict type system in Elixir reminds me of JavaScript. For instance, I wasted 30 minutes debugging xyzHtml versus xyzHTML.

Despite these challenges, my interest in the Phoenix Framework is still strong, especially in its innovative approach to web design and problem-solving. Ultimately, I hope to create something special that reflects all the knowledge I've gained.

Development Stage

This project is still a work in progress, with updates rolling out about every week. Right now, I'm focusing on getting the application up and running, so most of my efforts are pretty exploratory. I'm prioritizing the basics for now, putting off more complex features like relational stuff in Go templates and repositories. I've also stripped away some older code generators, like NodeJS Express and other read endpoints.

Pros

The organized structure makes it easy to navigate and maintain the code. Plus, being able to reuse code across different components really boosts productivity, saving time when you're developing new features. This setup is also great for scalability, meaning you can add new functionalities without messing up the existing code.

Cons

On the flip side, this structure can feel a bit overkill for smaller projects, which might add unnecessary complexity. There's also a learning curve to really get the hang of everything. And sometimes, it can lead to overengineering, where simpler solutions get overlooked in favor of more complicated ones, which can slow things down.

Example Breakdown

Here's a quick overview:

  • /cmd/mysite is the main entry point, specifically found in main.go.
  • /pkg holds reusable libraries:
    • /models/user.go defines the user model.
    • /services/user.go contains the business logic related to users.
    • /repositories/user.go manages database interactions and validates user-related data.
  • /internal includes the internal logic of the application:
    • /config is for application settings:
      • app.go handles application-level settings.
      • db.go takes care of database configurations.
    • /handlers manages HTTP endpoints:
      • /user has user-related handlers:
        • form.go handles form submissions for user data.
        • html.go takes care of server-side rendering for user pages.
        • json.go manages JSON endpoints for user data.
    • /middleware isn't in use yet, but it has example.go as a placeholder for future middleware (like authentication).
    • /routes outlines the application routes in routes.go.
  • /web contains frontend resources:
    • /templates for HTML templates:
      • /user has user-related templates:
        • show.html displays details for a single user.
        • new.html includes a form to create a new user.
        • index.html shows all users in a table.
        • edit.html features a form for editing user details.
    • /static contains static files like CSS and JS.
